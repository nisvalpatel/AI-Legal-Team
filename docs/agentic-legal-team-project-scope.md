# Agentic Legal Team — Broad Project Scope

## Project Overview

I am building an **agentic legal analysis platform** to learn how **multi-agent workflows** work in practice, especially for complex document-heavy tasks. The goal is to simulate a small “legal team” made up of specialized AI agents that collaborate on case analysis instead of relying on a single monolithic chatbot.

This system is **not intended to provide legal advice**. It is an educational and engineering-focused project designed to explore how multiple agents can coordinate, retrieve information, analyze evidence, and produce structured outputs with human review.

The platform should ingest legal case materials such as:

- police reports
- witness statements
- court filings
- transcripts
- evidence summaries
- audio/video transcripts
- public legal research sources

It should then use a set of specialized agents to:

- extract facts from documents
- build timelines of events
- identify contradictions across sources
- surface missing information or evidentiary gaps
- perform citation-grounded research over public legal databases
- generate structured strategy memos and review-ready summaries

The broader learning objective is to understand:

- how to decompose a complex problem into **specialized AI agents**
- how to orchestrate those agents with a workflow engine
- when to use **tool calls**, **shared state**, and **human-in-the-loop review**
- how retrieval, memory, and structured outputs affect multi-agent performance
- how to build a production-like system using modern backend, frontend, and infrastructure tools

---

## Core Problem the System Solves

Legal case analysis is inherently **multi-step, document-heavy, and cross-referential**. Important details are often spread across many disconnected documents and media sources. A single investigator, attorney, or analyst has to do all of the following:

- read and interpret large volumes of text
- extract relevant facts and entities
- compare competing narratives
- reconstruct event timelines
- identify contradictions or inconsistencies
- search legal authorities and precedents
- connect facts to legal standards
- prepare human-readable summaries or memos

This makes the domain a strong fit for a **multi-agent architecture** because the work naturally separates into roles.

Instead of one general-purpose assistant doing everything, the system should model a collaborative legal workflow where different agents each handle a focused task:

- an **Evidence Agent** processes files and extracts facts, entities, exhibits, and source-level claims
- a **Research Agent** retrieves statutes, cases, and public legal materials with citations
- a **Strategy Agent** synthesizes the factual record and research into arguments, issues, and next-step recommendations
- optionally, a **Timeline Agent**, **Contradiction Agent**, or **Review Agent** can be added for further specialization

The system should demonstrate how coordinated agents can outperform a single agent by:

- reducing context overload
- improving separation of responsibilities
- enabling tool-specific routing
- preserving traceability to source evidence
- supporting human oversight at key checkpoints

## Product Vision

The product is a **case analysis workbench** where a user uploads a set of legal case materials and receives a structured, reviewable analysis generated by multiple collaborating agents.

### Example user flow

1. The user uploads case files (PDFs, DOCX, images, transcripts, audio/video-derived text).
2. OCR and text extraction convert those sources into machine-readable documents.
3. Documents are chunked, embedded, indexed, and stored for retrieval.
4. Agents collaborate using a graph-based orchestration engine:
   - extract entities, dates, claims, and events
   - build a timeline
   - identify contradictions across testimony and reports
   - retrieve relevant legal authorities
   - generate a strategy memo with citations
5. The system presents outputs in a React UI.
6. A human reviewer can approve, edit, reject, or request re-analysis.
7. The final output remains traceable to original source documents and retrieval citations.

## Functional Goals

### A. Document Ingestion

The system should accept multiple file types and normalize them into a common internal representation.

#### Inputs

- PDF
- DOCX
- TXT
- images (PNG/JPG)
- OCR-processed scans
- transcript text
- optionally audio files for speech-to-text

#### Output

A structured internal document object containing:

- document id
- case id
- source file metadata
- extracted text
- page / section references
- OCR confidence (if applicable)
- ingestion status

---

### B. Retrieval-Augmented Legal Research

The platform should support **citation-grounded retrieval** using both semantic and keyword search.

#### Research data sources

- public case law collections
- public statutes / regulations
- legislative sources
- open legal information repositories

#### Retrieval requirements

- semantic vector search over indexed content
- keyword / lexical search for exact legal terms
- hybrid ranking to combine both approaches
- citation retention for every retrieved result
- source metadata stored for explainability

#### Goal

Ensure the system can answer:

- “What cases discuss this issue?”
- “What legal standards are relevant?”
- “What statutes or precedents match these facts?”
- “What are the most relevant authorities with citations?”

---

### C. Multi-Agent Analysis

The central learning objective is the orchestration of multiple specialized agents.

#### Example agent roles

##### 1. Evidence Agent

Responsible for:

- extracting factual claims from case files
- identifying people, places, dates, organizations, and evidence items
- summarizing each document
- converting unstructured materials into structured evidence records

##### 2. Timeline Agent

Responsible for:

- ordering events chronologically
- resolving relative or ambiguous dates where possible
- linking events back to source documents
- flagging time conflicts or missing intervals

##### 3. Contradiction Agent

Responsible for:

- comparing claims across documents
- identifying inconsistent statements
- highlighting factual disputes
- assigning confidence and source citations

##### 4. Research Agent

Responsible for:

- querying the legal knowledge base
- retrieving relevant cases/statutes
- returning results with citations and brief relevance rationales

##### 5. Strategy Agent

Responsible for:

- synthesizing evidence and research
- outlining key issues, strengths, weaknesses, and unanswered questions
- producing a review memo or litigation/investigation summary
- avoiding unsupported conclusions

##### 6. Human Review Agent / Review Step

Responsible for:

- pausing execution when human approval is needed
- enabling edits and overrides
- tracking approved vs. draft outputs

---

### D. Human-in-the-Loop Review

Since legal analysis is high-stakes, the system should include intentional checkpoints where a human reviewer can intervene.

#### Review actions

- approve extracted facts
- correct OCR output
- merge or split entities
- confirm chronology
- accept/reject contradiction findings
- validate final memo drafts

#### Why this matters

This ensures the project reflects real-world workflow design:

- AI supports analysis
- humans remain accountable for interpretation and final use

## Technical Scope

### Backend

#### Python + FastAPI

FastAPI should serve as the main application backend for:

- file upload APIs
- document ingestion pipelines
- OCR/transcription job endpoints
- agent orchestration endpoints
- retrieval/query endpoints
- review workflow APIs
- authentication/session hooks if added later

#### LangGraph

LangGraph should orchestrate the multi-agent workflow:

- define graph nodes for each specialized agent
- manage execution order
- preserve shared state between agents
- support conditional routing
- enable retries / fallback nodes
- allow interruption for human review

#### MCP (Model Context Protocol)

MCP can be used to expose tools and resources cleanly to the agents:

- retrieval tool
- document search tool
- citation lookup tool
- evidence browser
- timeline/query tools
- structured DB access tools

This helps make agents more modular and tool-driven rather than embedding everything directly into prompts.

---

### Data Layer

#### PostgreSQL

PostgreSQL acts as the system-of-record for:

- cases
- uploaded documents
- extracted entities
- evidence items
- timelines
- contradictions
- research sessions
- user annotations
- review decisions

#### pgvector and/or FAISS

Used for semantic search over:

- uploaded case materials
- legal research corpora
- summaries and extracted claims

#### ElasticSearch

Used for:

- keyword search
- exact match queries
- legal phrase search
- hybrid retrieval with vector search

This allows the system to support both:

- semantic similarity
- precision retrieval for citations, names, statutory text, and key phrases

---

### Frontend

#### ReactJS

The frontend should provide a case workspace where users can:

- upload files
- view ingestion progress
- inspect OCR/transcription output
- search documents and legal sources
- review extracted facts and entities
- inspect timeline entries
- examine contradictions with source citations
- approve or edit strategy memos

#### Key UI modules

- Case Dashboard
- Document Explorer
- Timeline Viewer
- Contradiction Review Panel
- Research Results Panel
- Memo / Strategy Output Panel
- Human Review Queue

---

### OCR / Text Extraction

#### Tesseract OCR

Used for scanned or image-based documents to extract text from:

- exhibits
- scans
- handwritten-adjacent or poor-quality documents where possible

Optionally, a speech-to-text stage can process audio/video-derived transcripts if included later.

---

### Infrastructure

#### Docker

Every service should be containerized for consistency and reproducibility.

#### Kubernetes

Used to simulate production-scale deployment and learn service orchestration concepts such as:

- multi-service deployment
- scaling workers
- job queues
- persistent volumes
- service discovery
- environment configuration
- isolated processing workloads

#### Linux + Git

Linux should be the target development/runtime environment, and Git should be used for:

- source control
- branching strategy
- collaboration and experiment tracking

## Expected Outputs

The system should generate structured outputs that are useful for inspection and debugging, not just freeform text.

### Output types

- document summaries
- extracted entities
- evidence tables
- event timelines
- contradiction reports
- research result lists with citations
- strategy memos
- final review packets

### Important design principle

Every major output should be:

- **traceable**
- **citation-grounded**
- **reviewable**
- **stored as structured data where possible**

---

## Constraints and Non-Goals

To keep the project focused:

### Constraints

- this is a learning project first, not a production legal product
- outputs must be treated as decision-support, not legal advice
- hallucination risk should be minimized through citations and source linking
- human review should remain part of the workflow
- modular architecture is preferred over premature optimization

### Non-goals (for the first version)

- replacing attorneys
- fully autonomous legal strategy generation without review
- handling every legal domain equally well
- supporting all file formats from day one
- solving perfect OCR or perfect contradiction detection initially
